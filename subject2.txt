	-------------------------------------------------Ներածություն-------------------------------------------------
	Այս նախագիծը նպատակ ունի ընդլայնել ձեր գիտելիքները համակարգային ադմինիստրացիայի մասին՝ օգտագործելով Docker: Դուք կվիրտուալացնեք մի քանի Docker պատկերներ՝ ստեղծելով դրանք ձեր նոր անձնական վիրտուալ մեքենայում:
	
	-------------------------------------------------Ընդհանուր ուղեցույցներ-------------------------------------------------
	
	• Այս նախագիծը պետք է իրականացվի Վիրտուալ Մեքենայի վրա:
	• Ձեր նախագծի կոնֆիգուրացիայի համար անհրաժեշտ բոլոր ֆայլերը պետք է տեղադրվեն srcs պանակում:
	• Անհրաժեշտ է նաև Makefile, որը պետք է գտնվի ձեր պանակի արմատում: Այն պետք է կարգավորի ձեր ամբողջ ծրագիրը (այսինքն՝ այն պետք է կառուցի Docker պատկերները՝ 		օգտագործելով docker-compose.yml):
	• Այս առարկան պահանջում է պրակտիկայում կիրառել հասկացություններ, որոնք՝ կախված ձեր ֆոնից, դուք դեռ չեք սովորել: Հետևաբար, մենք խորհուրդ ենք տալիս չվարանել կարդալ Docker-ի օգտագործմանը վերաբերող շատ փաստաթղթեր, ինչպես նաև ցանկացած այլ բան, որը կգտնեք օգտակար այս առաջադրանքը կատարելու համար:
	
	-------------------------------------------------Պարտադիր մաս-------------------------------------------------
	
	Այս նախագիծը կազմված է այն բանից, որ դուք կարգավորեք փոքր ենթակառուցվածք, որը բաղկացած է տարբեր ծառայություններից՝ որոշակի կանոնների համաձայն: Ամբողջ նախագիծը պետք է արվի վիրտուալ մեքենայում: Դուք պետք է օգտագործեք docker compose:
	Ամեն Docker պատկերը պետք է ունենա նույն անունը, ինչ իր համապատասխան ծառայությունը: Ամեն ծառայություն պետք է աշխատի նվիրված կոնտեյներում: Արտադրողականության պատճառով՝ կոնտեյներները պետք է կառուցվեն կամ Alpine-ի, կամ Debian-ի նախավերջին կայուն տարբերակից: Ընտրությունը ձերն է:
	Դուք նաև պետք է գրեք ձեր սեփական Dockerfile-ները՝ մեկը յուրաքանչյուր ծառայության համար: Dockerfile-ները պետք է կանչվեն ձեր docker-compose.yml-ում ձեր Makefile-ի կողմից: Սա նշանակում է, որ դուք պետք է ինքներդ կառուցեք ձեր նախագծի Docker պատկերները: Այդ պատճառով արգելվում է պատրաստի Docker պատկերները քաշել (pull), ինչպես նաև օգտագործել այնպիսի ծառայություններ, ինչպիսիք են DockerHub-ը (Alpine/Debian-ը բացառվում են այս կանոնից):
	Ապա դուք պետք է կարգավորեք՝
	
		• Docker կոնտեյներ, որը պարունակում է NGINX՝ միայն TLSv1.2 կամ TLSv1.3-ով:
		• Docker կոնտեյներ, որը պարունակում է WordPress + php-fpm (այն պետք է տեղադրված և կարգավորված լինի) միայն՝ առանց nginx-ի:
		• Docker կոնտեյներ, որը պարունակում է MariaDB միայն՝ առանց nginx-ի:
		• Հատոր (volume), որը պարունակում է ձեր WordPress տվյալների բազան:
		• Երկրորդ հատոր, որը պարունակում է ձեր WordPress կայքի ֆայլերը:
		• Դուք պետք է օգտագործեք Docker-ի անվանված հատորներ (named volumes) այս երկու մշտական պահեստավորման համար: Bind mounts-ը արգելված է այս հատորների համար:
		• Երկու անվանված հատորներն էլ պետք է իրենց տվյալները պահեն /home/login/data-ում հոսթ մեքենայի վրա: Փոխարինեք "login"-ը ձեր ուսանողի օգտանվամբ:
		• Docker-ցանց (docker-network), որը հաստատում է կապը ձեր կոնտեյներների միջև:
			Ձեր կոնտեյներները պետք է վերագործարկվեն վթարի դեպքում:

		ℹ️ Docker կոնտեյները վիրտուալ մեքենա չէ: Այդ պատճառով խորհուրդ չի տրվում օգտագործել ցանկացած խորամանկ կարկատան (hacky patches), որը հիմնված է 'tail -f'-ի և նմանատիպ մեթոդների վրա՝ այն գործարկելու փորձի ժամանակ: Կարդացեք, թե ինչպես են աշխատում դեմոնները (daemons) և արդյոք դրանք օգտագործելը լավ գաղափար է, թե ոչ:
	
		❗ Իհարկե, network: host կամ --link կամ links:-ի օգտագործումը արգելված է: network տողը պետք է ներկա լինի ձեր docker-compose.yml ֆայլում: Ձեր կոնտեյներները չպետք է գործարկվեն անվերջ ցիկլ աշխատացնող հրամանով: Այդ պատճառով սա նաև վերաբերում է ցանկացած հրամանի, որը օգտագործվում է որպես entrypoint կամ entrypoint-սկրիպտներում: Հետևյալներն արգելված խորամանկ կարկատաններից մի քանիսն են՝ tail -f, bash, sleep infinity, while true:
		
		ℹ️ Կարդացեք PID 1-ի և Dockerfiles գրելու լավագույն փորձի մասին:
			• Ձեր WordPress տվյալների բազայում պետք է լինեն երկու օգտատեր, որոնցից մեկը ադմինիստրատորն է: Ադմինիստրատորի օգտանունը չի կարող պարունակել admin/Admin կամ administrator/Administrator (օրինակ՝ admin, administrator, Administrator, admin-123 և այլն):
		ℹ️ Ձեր հատորները հասանելի կլինեն /home/login/data պանակում՝ հոսթ մեքենայի վրա՝ օգտագործելով Docker: Իհարկե, դուք պետք է փոխարինեք login-ը ձերով:
			Բաներն ավելի պարզ դարձնելու համար՝ դուք պետք է կարգավորեք ձեր դոմենի անունը, որպեսզի այն ուղղորդի ձեր տեղական IP հասցեին: Այս դոմենի անունը պետք է լինի login.42.fr: Կրկին, դուք պետք է օգտագործեք ձեր սեփական login-ը: Օրինակ, եթե ձեր login-ը wil է, ապա wil.42.fr-ը կվերահղի դեպի IP հասցեն, որը ցույց է տալիս wil-ի կայքը:
		
		❗ latest թեգն արգելված է: Ոչ մի գաղտնաբառ չպետք է լինի ձեր Dockerfile-ներում: Պարտադիր է օգտագործել միջավայրի փոփոխականներ (environment variables): Նաև պարտադիր է օգտագործել .env ֆայլ՝ միջավայրի փոփոխականները պահելու համար: Ինտենսիվորեն խորհուրդ է տրվում օգտագործել Docker secrets՝ ցանկացած գաղտնի տեղեկատվությունը պահելու համար: Ցանկացած հավատարմագրեր (credentials), API բանալիներ կամ գաղտնաբառեր, որոնք գտնվում են ձեր Git պահոցում (պրոպեր կարգավորված secrets-ներից դուրս), կհանգեցնեն նախագծի ձախողման: Ձեր NGINX կոնտեյները պետք է լինի ենթակառուցվածք մուտք գործելու միակ մուտքը՝ միայն 443 պորտի միջոցով՝ օգտագործելով TLSv1.2 կամ TLSv1.3 պրոտոկոլը:
